# Evaluate Voxtral Realtime inference options for 16GB VRAM

## Context

vLLM serving Voxtral-Mini-4B-Realtime-2602 in bf16 on the RTX 5060 Ti 16GB
leaves only ~4.5GB for KV cache (~2304 tokens). At 12.5 audio tokens/sec,
that's ~3 minutes before saturation. The model has sliding window attention
(text: 8192, audio: 750) but vLLM doesn't evict KV blocks outside the window,
so "realtime" streaming dies after the cache fills.

Current state: vLLM is running with max-model-len=8192 and works for short
bursts but not continuous streaming. Need to evaluate alternatives.

## Options to evaluate

### Quantized models
- **VQF quantized**: https://huggingface.co/tantk/Voxtral-4B-Realtime-VQF
- **INT4 Jetson build**: https://huggingface.co/Teaspoon-AI/Voxtral-Mini-4B-INT4-Jetson
- MLX quantizations exist but irrelevant (no Apple Silicon)

### Alternative runtimes
- **voxtral.cpp** (llama.cpp fork): https://github.com/andrijdavid/voxtral.cpp
  - Pure C inference, likely handles sliding window natively
  - No vLLM overhead, direct CUDA, probably much better VRAM efficiency
  - May actually do what "realtime" promises on 16GB

### Key questions
1. Does voxtral.cpp handle sliding window eviction properly?
2. What's the VRAM footprint of voxtral.cpp with Q4 quantization?
3. Can we get indefinite streaming on 16GB with any of these?
4. What's the latency/accuracy tradeoff vs bf16 vLLM?
