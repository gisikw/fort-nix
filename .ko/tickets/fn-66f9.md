# Automated sops rekey with derived .sops.yaml

Automated .sops.yaml generation and rekey, replacing the CI rekeying pipeline.

## Problem

Current secrets workflow:
- secrets.nix keys everything to principals only
- CI (release.yml) evals every host's config.age.secrets, derives per-host recipients, decrypts with CI key, re-encrypts for consumers
- This is ~220 lines of CI that exists because secrets.nix can't express 'whoever uses this'

## Design

A just rekey command that:
1. Walks nixosConfigurations.* from the flake
2. For each host, collects which sops secrets it references
3. Derives the host's device pubkey from the cluster manifest
4. Generates .sops.yaml creation rules (secret path → list of recipient keys)
5. Runs sops updatekeys on any secrets whose rules changed

.sops.yaml is a generated artifact — do not edit manually. Add a header comment and a pre-commit check that rejects hand-edited .sops.yaml.

## Commands

- just rekey — full derivation + rekey all secrets
- just rekey <path> — rekey a single secret (skip derivation, use existing .sops.yaml)
- Pre-commit hook: if any .sops secret files are staged, run derivation, verify .sops.yaml is up to date, rekey any newly-added secrets

## Performance

The CI version evals every host serially. Optimize:
- Parallel host eval (nix eval is the bottleneck)
- Cache the recipients map; only re-derive when host manifests or app imports change
- Incremental rekey: only touch secrets whose recipient list or content changed

## Dependencies

- fn-e5a4 (validate sops-nix works at all first)

## Success criteria

- just rekey produces a correct .sops.yaml from flake eval
- Pre-commit hook catches un-rekeyed secrets
- CI release.yml rekeying steps can be removed
- release branch concept can be eliminated (main is deployable)
