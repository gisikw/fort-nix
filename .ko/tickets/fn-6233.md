# Build claude-spawn daemon (clauded) for process-isolated claude invocation

## Problem

Claude Code suppresses all Bash tool output when any child `claude` process exists in the process tree. This affects automation invoked from within claude sessions (daily briefing, ko agent, ad-hoc spawns). The suppression is parent-side -- output IS produced (verified via file redirects) but the Bash tool returns empty results.

Current `claude-spawn` wrapper (writeShellScriptBin in dev-sandbox) handles env var stripping (CLAUDECODE, CLAUDE_CODE_ENTRYPOINT) and works for automation that manages its own output, but inline Bash tool usage gets blanked.

## Proposal

A persistent daemon (`clauded`) that:
- Runs as its own process tree (fully isolated from any claude session)
- Accepts requests via unix socket (args + stdin payload)
- Spawns `claude` in a clean environment
- Returns stdout/stderr/exit code to the caller
- Lightweight -- no queuing, no concurrency control (initially)

## What we know

- `unset CLAUDECODE CLAUDE_CODE_ENTRYPOINT` is sufficient for claude to run
- The output suppression is NOT env-var-based, NOT process-name-based
- Renaming the binary, setsid, nohup, /dev/null redirects -- none bypass it
- The parent monitors the process tree somehow (possibly /proc or session state)
- A truly separate process tree (like a systemd service) would bypass it entirely
- Knockout double-fork daemonization (`_daemonize` subcommand) is prior art

## Implementation notes

- Go binary, systemd service in dev-sandbox aspect
- CLI client: `clauded -p "prompt"` mirrors claude interface
- Socket at /run/user/1000/clauded.sock or similar
- Existing `claude-spawn` wrapper stays for simple automation use cases
